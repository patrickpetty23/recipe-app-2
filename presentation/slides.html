Success & Failure Planning — 9H

Did it change what users do?

Success = behavior changed. Technical performance is a floor, not the goal.
Behavioral outcomes — what mattered
Question Evidence
Full workflow completed unprompted? Thomas: scan → list → live Walmart cart ✅
User described the feature before seeing it? "It'd be cool if it added to my Walmart order" — Thomas, Round 4 ✅
Feature emerged from listening, not the PRD? Nutrition tracker — not in v1.0, most-engaged by final round ✅
User showed it to someone else? Thomas showed his wife on the spot — "I'm sold" ✅
Second segment discovered through use? Sherrie — in-store shopper, different job than assumed ✅
Technical floor — needs to be good enough not to block behavior
Ingredient accuracy 94% avg (target 90%+) ✅
Camera latency 14s avg (target <10s) — non-blocking UI mitigates ⚠️
App stability 0 crashes across all test sessions ✅
Where behavior was blocked

Per-serving logging defaulted to all servings — Thomas found it. This breaks the tracker loop: if the data is wrong, users stop logging.

Walmart price off ~$6 — Kierra: "I'd prefer no price if it's going to change." Inaccurate data erodes trust before behavior can form.

Camera at 14s didn't block anyone in testing, but it's the only technical miss that could stop a first-time user from continuing.

The real lesson

Speed was never the value driver. Nobody chose this app because it was fast. They chose it because it finished a job they couldn't finish before — recipe → ingredients → shopping cart in one tap.

The best signal we got: a user described a feature we hadn't shown yet, in their own words. That doesn't come from hitting a latency target.

What We Learned

Honest reflection

What surprised us

1

The midterm "What's Next" column became the roadmap — everything shipped

URL import, native app, Walmart, ingredient substitutes — all five items delivered. The column was the plan.

2

URL import was 3× more common than camera

Changed our P0 priority and the home screen design. We had it backwards.

3

Nutrition tracking was not in the PRD

Built entirely from user feedback. Became the most-engaged feature. Listening works.

4

Thomas validated Walmart cart before seeing it

Described the exact feature unprompted. Strongest possible product validation signal.

What we'd do differently

  • Fix per-serving logging bug before demo — affects Tracker trust
  • Camera speed: show optimistic UI earlier to mask API latency
  • Build export-to-Notes for Sherrie's user type from the start

Pivot plan

If Walmart integration doesn't find traction: build export-to-Notes as the alternative exit path for in-store shoppers. The core product (AI extraction + cooking mode + nutrition tracking) remains valuable either way.
